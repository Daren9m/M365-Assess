# v1.6.0 Value Opportunity Design Spec

**Date:** 2026-04-03
**Milestone:** v1.6.0 - Value Opportunity
**Scope:** 8 issues (#295-#302)

## Context

Organizations pay for Microsoft 365 licenses but often underutilize the features included. The Value Opportunity section analyzes license utilization and feature adoption during the assessment run, producing an actionable roadmap of features the tenant is paying for but not using. This turns the assessment from a security audit into a strategic advisory tool.

The architecture is passive: an adoption signal accumulator captures data as existing collectors run (zero new API calls, zero collector changes), then three post-hoc collectors analyze the signals against a feature taxonomy. A single integrated report page presents the results.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| SKU feature map | Created from scratch in CheckID, synced | No existing map; needs domain-specific design |
| Audience | Both consultants and internal IT | Neutral language works for both |
| Signal source | Combine Add-Setting results + CSV stats | Binary detection + depth metrics |
| Report output | Integrated page in main report sidebar | Consistent UX, no standalone exports |
| Feature granularity | Sub-feature level (~40-60 features) | Actionable without drowning in noise |
| Effort model | Static metadata in feature map | Simple, consistent, maintainable |
| Architecture | Passive accumulator + post-hoc analysis | Zero API calls, zero collector changes |

## Implementation Order

```
#302 (sku-feature-map.json) -- data foundation, no code dependencies
#295 (adoption signal accumulator) -- SecurityConfigHelper hook
#296 (section registration) -- AssessmentMaps + wizard + orchestrator
#297 (license utilization collector)
#298 (feature adoption collector) -- depends on #295, #302
#299 (feature readiness collector) -- depends on #297, #298
```

### Field Reference

| Field | Type | Description |
|-------|------|-------------|
| `featureId` | string | Unique kebab-case identifier |
| `name` | string | Display name |
| `category` | string | References `categories[].id` |
| `description` | string | One-line description for report |
| `effortTier` | enum | `Quick Win`, `Medium`, `Strategic` |
| `requiredServicePlans` | string[] | Service plan names from `Get-MgSubscribedSku` (OR logic) |
| `checkIds` | string[] | Base CheckIds (without sub-numbers) that indicate this feature |
| `csvSignals` | object[] | Optional CSV-based depth metrics |
| `csvSignals[].file` | string | CSV filename pattern (e.g., `07b-Entra-Security-Config.csv`) |
| `csvSignals[].column` | string | Column to read |
| `csvSignals[].match` | string | Wildcard pattern to match rows |
| `csvSignals[].metric` | string | Metric type: `passRate` (pass/total from Status column), `count` (row count matching pattern), `percentage` (numeric value from matched column) |
| `prerequisites` | string[] | featureIds that must be adopted first |
| `learnUrl` | string | Microsoft Learn documentation URL |
| `tags` | string[] | Searchable labels (E3, E5, Security, Compliance, etc.) |

## 2. Adoption Signal Accumulator (#295)

~15-line addition to `SecurityConfigHelper.ps1`. Zero changes to existing collectors.

### Changes to SecurityConfigHelper.ps1

**Initialize-SecurityConfig:** Add `$script:AdoptionSignals = @{}` initialization.

**Add-SecuritySetting:** After the existing object is appended to `$Settings` (line ~116), add:
```powershell
$script:AdoptionSignals[$subCheckId] = @{
    Status       = $Status
    Setting      = $Setting
    CurrentValue = $CurrentValue
    Category     = $Category
}
```

**New function: Get-AdoptionSignals**
```powershell
function Get-AdoptionSignals {
    [CmdletBinding()]
    [OutputType([hashtable])]
    param()
    if ($script:AdoptionSignals) {
        return $script:AdoptionSignals.Clone()
    }
    return @{}
}
```

### Data Shape

```powershell
@{
    'ENTRA-PIM-001.1' = @{ Status = 'Fail'; Setting = 'PIM Enabled'; CurrentValue = 'False'; Category = 'PIM' }
    'ENTRA-PIM-002.1' = @{ Status = 'Pass'; Setting = 'Role Activation'; CurrentValue = 'Configured'; Category = 'PIM' }
    'CA-MFA-ALL-001.1' = @{ Status = 'Pass'; Setting = 'MFA for All Users'; CurrentValue = 'Yes (1 policy)'; Category = 'Conditional Access' }
    # ... one entry per Add-Setting call across all collectors
}
```

### Constraints

- No changes to existing collector code
- Signals accumulate passively as `Add-Setting` is called
- Clone on read for thread safety
- Signals are available to Value Opportunity collectors which run last

## 4. Three Collectors (#297, #298, #299)

All follow the data-collector pattern: PSCustomObject output + CSV export. All run in the ValueOpportunity section (last position).

### 4a. Get-LicenseUtilization.ps1 (#297)

**Input:**
- `Resolve-TenantLicenses` output (passed via parameter or re-invoked)
- `sku-feature-map.json` (loaded from controls/)

**Output columns:**
| Column | Type | Description |
|--------|------|-------------|
| FeatureId | string | Feature identifier |
| FeatureName | string | Display name |
| Category | string | Category name |
| IsLicensed | bool | Tenant has required service plan |
| SourceSkus | string | Comma-separated SKU part numbers that provide this feature |
| LicensedUserCount | int | Users assigned to the source SKU (approximate) |

**Logic:**
```
For each feature in sku-feature-map.json:
  Check if ANY requiredServicePlan is in ActiveServicePlans
  If yes: IsLicensed = true, find source SKUs
  If no: IsLicensed = false
```

### 4b. Get-FeatureAdoption.ps1 (#298)

**Input:**
- `Get-AdoptionSignals` output
- CSV files from assessment folder
- `sku-feature-map.json`
- License utilization output (for NotLicensed state)

**Output columns:**
| Column | Type | Description |
|--------|------|-------------|
| FeatureId | string | Feature identifier |
| FeatureName | string | Display name |
| Category | string | Category name |
| AdoptionState | enum | Adopted, Partial, NotAdopted, Unknown, NotLicensed |
| AdoptionScore | int | 0-100 based on check pass rate |
| PassedChecks | int | Number of checks that passed |
| TotalChecks | int | Number of checks that ran |
| DepthMetric | string | Optional metric from CSV (e.g., "47/52 users with MFA") |

**Scoring logic:**
```
For each feature:
  If not licensed -> NotLicensed
  Map feature.checkIds to adoption signals (match base CheckId prefix)
  Count passed vs total
  If no signals found -> Unknown
  If all pass -> Adopted (100)
  If some pass -> Partial (proportional 1-99)
  If all fail -> NotAdopted (0)

  If csvSignals defined:
    Parse referenced CSV, compute depth metric
    Append to DepthMetric column
```

### 4c. Get-FeatureReadiness.ps1 (#299)

**Input:**
- License utilization output
- Feature adoption output
- `sku-feature-map.json`

**Output columns:**
| Column | Type | Description |
|--------|------|-------------|
| FeatureId | string | Feature identifier |
| FeatureName | string | Display name |
| ReadinessState | enum | Ready, Blocked, NotLicensed |
| Blockers | string | Comma-separated blocker descriptions |
| EffortTier | string | Quick Win, Medium, Strategic |
| LearnUrl | string | Microsoft Learn link |

**Logic:**
```
For each feature:
  If not licensed -> NotLicensed, blocker = "Requires [plan names]"
  If prerequisites not adopted -> Blocked, blocker = "Requires [prereq names]"
  Else -> Ready
```

## 5. Analysis Engine (#300)

`ValueOpportunity/Analyze-ValueOpportunity.ps1` -- merges three collector outputs.

**Input:** License utilization, feature adoption, feature readiness data + sku-feature-map.json
**Output:** Hashtable with:

```powershell
@{
    OverallAdoptionPct = 67          # % of licensed features that are Adopted/Partial
    LicensedFeatureCount = 42       # features the tenant is paying for
    AdoptedFeatureCount = 28        # features fully adopted
    GapCount = 14                   # licensed but not adopted
    CategoryBreakdown = @(          # per-category stats
        @{ Category = 'Identity & Access'; Licensed = 12; Adopted = 9; Pct = 75 }
        @{ Category = 'Email Security'; Licensed = 8; Adopted = 5; Pct = 63 }
        # ...
    )
    Roadmap = @{
        QuickWins  = @( ... )       # licensed + not adopted + effort = Quick Win
        Medium     = @( ... )       # licensed + not adopted + effort = Medium
        Strategic  = @( ... )       # licensed + not adopted + effort = Strategic
    }
    GapMatrix = @(                  # category x adoption state cross-reference
        @{ Category = 'Identity & Access'; Adopted = 9; Partial = 2; NotAdopted = 1; Unknown = 0 }
        # ...
    )
    NotLicensed = @( ... )          # features not licensed (upsell opportunity)
}
```

## 6. Report Page (#301)

Single integrated page in the sidebar: **"Value Opportunity"** -- positioned between Compliance Overview and Technical Issues.

### Layout (top to bottom)

**1. Hero Panel**
- Donut chart: overall adoption percentage (reuses `Get-SvgDonut`)
- Three stat cards: Licensed Features / Adopted / Gaps
- One-line summary: "Your organization uses 28 of 42 licensed features (67%)"

**2. Category Breakdown**
- Horizontal stacked bars per category (Adopted / Partial / Not Adopted)
- Color coding: green (adopted), amber (partial), red (not adopted)

**3. Quick Wins Table**
- Sortable table of licensed-but-not-adopted features with effort = Quick Win
- Columns: Feature, Category, Adoption %, MS Learn link
- Call-to-action: "These features are already licensed and require minimal effort to enable"

**4. Full Roadmap (collapsible)**
- Three `<details>` sections: Quick Wins / Medium Effort / Strategic
- Each contains a feature table with: Feature, Category, Adoption State, Score, Effort, Blockers, Learn URL

**5. Gap Matrix**
- Grid/heatmap: categories on rows, adoption states on columns
- Cell values: feature count
- Visual: darker shading for higher counts

### CSS/Theme

Reuses existing CSS variables (`--m365a-accent`, `--m365a-card-bg`, etc.). New classes:
- `.value-hero` -- hero panel container
- `.value-stat-card` -- stat cards (mirrors existing `.stat-card` pattern)
- `.value-category-bar` -- category breakdown bar
- `.value-roadmap` -- roadmap section
- `.value-gap-matrix` -- gap matrix grid

### Build Integration

New function in `Build-SectionHtml.ps1` or new file `Build-ValueOpportunityHtml.ps1` that:
- Runs `Analyze-ValueOpportunity` to get the analysis model
- Renders the HTML sections
- Called from `Get-ReportTemplate.ps1` when ValueOpportunity data exists
